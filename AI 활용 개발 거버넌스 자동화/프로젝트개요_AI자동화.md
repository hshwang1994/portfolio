# AI 기반 개발·변경 검증 자동화 — 프로젝트 개요

> 이 문서는 하네스 프로젝트 **전체**에 대한 개요입니다 (개별 문제 해결은 `문제해결_8선_AI자동화.md` 참고).
> 모든 수치는 코드·문서를 **직접 센 '규모' 값**(파일 개수, 커밋 비율 등 / 2026-06-06 기준)입니다. 리뷰 시간·비용·토큰 절감 같은 **효과(ROI)는 측정 체계가 없어 수치로 넣지 않았습니다.**

---

## 추진 배경

자사의 클라우드 제품은 OpenStack, VMware, Kubernetes, Ansible 등을 통합 관리하는 프라이빗 클라우드 플랫폼입니다. 이 제품은 고객사별로 완전히 별도 코드베이스를 운영하는 구조가 아니라, **하나의 공통 코드베이스**를 여러 고객사가 함께 쓰고, 고객사별 차이는 **고객사 전용 확장 모듈(plugin)** 과 설정·연동 방식으로 분리하는 구조였습니다(예: 한 고객사만 쓰는 기능은 공통 코드가 아니라 그 고객사 전용 모듈에 둠).

고객사가 늘어남에 따라 요구사항 또한 늘어만 갔지만, 개발 인력은 같은 속도로 늘릴 수 없었습니다. 그래서 Claude Code 같은 AI 개발 도구를 도입해 한 사람이 처리할 수 있는 작업량을 늘리고자 했습니다.

> *(그림: 요구사항 증가에 따른 대응)*

문제는 **AI가 코드를 잘 만들어도, 운영 맥락을 모르면 위험하다**는 점이었습니다. 예를 들어 외부 시스템과 연동할 때 실제로 주고받는 값, 고객사별로 분리되어야 하는 로직, 함부로 바꾸면 안 되는 배포 설정 파일, 운영 장애를 겪으며 정착된 코딩 규칙은 대부분 문서로 명확히 남아 있지 않고 사람이 머리로 알던 지식이었습니다.

이런 환경에서 AI가 만든 변경을 무작정 반영하면 곧바로 문제가 발생합니다.

1. 한 고객사용 분기가 공통 영역에 섞이면 다른 고객사들 배포에 영향이 갑니다.
2. 외부 시스템(Jenkins, VMware, GitLab 등) 실제 계약과 어긋난 코드가 들어가면 운영 사고로 노출됩니다.
3. 회사 코드 컨벤션을 따르지 않으면 다른 사람이 유지보수하기 어려워집니다.
4. 이전에 발견된 오류도 사람 기억에만 의존하면 다시 반복됩니다.

그래서 이 프로젝트의 목적은 **AI가 만든 변경이 제품에 들어오기 전에 영향 범위·고객사 경계·외부 시스템 계약·회사 컨벤션을 확인하게 만들고, 한 번 발견한 오류는 다음부터 다시 발생하지 않게 만드는 것**이었습니다.

## 프로젝트 정의

이 프로젝트에서 말하는 **하네스**는 단순한 Claude 설정 파일이 아니라, AI가 코드를 변경할 때 **어디까지 허용하고 어디서 멈춰야 하는지를 정한 개발 거버넌스**였습니다.

이 프로젝트는 세 가지 설계 원칙을 기준으로 만들었습니다.

1. 사람 머릿속에 있던 운영 지식을 **규칙과 정책 파일로 꺼내는 것**
2. 자동으로 잡을 수 있는 문제는 **코드를 저장하는 순간 자동 검사로 빠르게 막는 것**
3. 코드만 봐서는 알 수 없는 외부 시스템(Jenkins·VMware 등)의 실제 동작 방식은 AI가 추측하지 못하게 **사람에게 먼저 묻도록 하는 것**

> *(그림: 하네스 전체 구조)*

**첫째**, AI가 만든 코드도 사람이 짠 코드와 똑같이 **세 단계로 검사**하게 했습니다. ① 코드를 만들기 전에 영향 범위를 확인하고, ② 코드를 저장하는 순간 위험한 변경을 막고(비밀번호·키나 배포 설정처럼 치명적인 건 아예 막고, 한 고객사 코드가 공통 영역에 섞이는 것처럼 애매한 건 경고만), ③ 다른 사람 코드가 합쳐진 뒤 안전장치·규칙 준수를 다시 점검합니다.

**둘째**, 사람과 AI가 같은 기준으로 개발하고 검토할 수 있도록 규칙·정책·검사 코드·사례를 **하나의 검증 체계로 통합**했습니다.

**셋째**, 발견된 오류는 정책과 검사 코드에 반영해 **같은 실수가 반복되지 않도록** 했습니다.

## 프로젝트 결과

이 프로젝트로 사람 머릿속에만 있던 운영 지식을 **검사할 수 있는 형태로 꺼내고**, AI가 만든 변경을 사람 코드와 같은 기준으로 점검하는 체계를 운영하게 됐습니다.

### 정량 결과

운영 중 실제로 겪은 오류·불일치 **108건**(한 고객사 전용 코드가 공통 영역에 섞임, 외부 시스템 값 불일치, 보안 규칙 위반 등)을, 그때만 고치고 끝내지 않고 **다시는 같은 실수가 나지 않도록 자동 검사·규칙으로 바꿨습니다.** 사람 머릿속에만 있던 운영 지식을 꺼낸 결과물입니다.

- **규칙 문서 34개 + 정책 설정 12개** — 'AI가 지켜야 할 선'을, 사람이 보는 문서(34개)와 검사 프로그램이 바로 읽는 설정 파일(12개)로 정리했습니다. (예: "한 고객사만 쓰는 코드를 모두가 쓰는 공통 부분에 넣지 말 것", "비밀번호·키를 코드에 남기지 말 것")
- **자동 검사 코드 29개** — 위 규칙 중 자동으로 판단할 수 있는 것을, 코드를 저장하거나 합칠 때 프로그램이 알아서 검사합니다.
- **AI 작업 절차서 59개 + 역할별 AI 작업자 50개** — 사람과 AI가 같은 방식으로 일하도록 만든 협업 자산입니다.

검사를 무작정 늘리지 않았습니다. 과거에 문제가 됐던 위험 패턴 11가지 중 **자동 검사로 확실히 잡을 수 있는 4가지만 막고**(예: 미완성 표시(TODO)가 남은 코드, 빈 화면 메시지), **사람 판단이 필요한 7가지(코드 흐름을 깊이 봐야 하는 것들)는 사람이 직접 확인**하게 나눴습니다 — 무리하게 자동화하면 틀린 경고(헛알림)가 늘어 개발자가 결국 경고를 무시하기 때문입니다.

보안 정보(비밀번호, 외부 접속용 키 등)는 **9가지 형태로 자동으로 찾아내** 코드에 들어가기 전에 막고, 시스템 기록(로그)에 남더라도 **6가지 방식으로 가려** 외부로 노출되지 않게 했습니다.

이 검증 체계의 규칙·정책·검사 코드를 **본인이 직접 설계하고 구현**했습니다. 자동화 영역(`.claude/`, `scripts/ai/`)을 함께 손댄 여러 명 중 **본인이 가장 많은 변경을 작성한 주 담당자**입니다(구체적 작성 비율과 측정 범위는 포트폴리오 본문 참고).

### 정성 결과 — 무엇이 달라졌나

- **AI가 "이 정도면 괜찮겠지" 하고 멋대로 판단해 코드를 만드는 일을 막았습니다.** 코드를 만들기 전에 영향 범위부터 확인하게 했습니다.
- **위험을 발견하는 시점이 크게 앞당겨졌습니다.** 예전엔 코드 리뷰나 배포 뒤에야 드러나던 문제(비밀번호 노출, 한 고객사 코드가 공통 영역에 섞임 등)를, 이제는 **코드를 저장·합치는 순간** 자동으로 잡습니다. 특히 다른 사람 코드가 합쳐질 때는 **7가지 위험을 자동 점검한 보고서**를 남깁니다.
- **AI에 들어가는 정보량(비용)을 줄였습니다.** 매번 프로젝트 상태(데이터베이스 구조·기능 목록 등)를 처음부터 다 읽지 않고, 정보마다 '언제 다시 확인할지'를 정해 **자주 바뀌는 것만 다시 읽게** 했습니다(현재 12가지 정보에 적용).
- **코드만 봐서는 알 수 없는 외부 시스템(Jenkins·VMware 등)의 동작 방식은, AI가 추측하지 않고 사람에게 먼저 묻도록** 했습니다. 한 대표 사례에서는 원인을 찾는 시도가 **3번에서 1번으로** 줄었습니다(사례 1건 기준).
- **한 번 겪은 오류가 자동 검사·규칙으로 남아**, 다음 개발자나 다음 AI 세션이 같은 실수를 반복하지 않습니다.
- **검증의 기준 자체를 슬쩍 풀어 약하게 만드는 일을 막았습니다.** 보안 규칙이나 고객사 경계 같은 '무엇을 검사할지 정하는 설정'을 바꾸려면 반드시 사람이 직접 승인하게 해서, AI가 스스로 자기 제약을 풀어버리지 못하게 했습니다.

### 한계 (정직하게)

- 리뷰 시간·비용·토큰 절감 같은 **효과(ROI)는 측정 체계가 없어 수치로 제시하지 않았습니다.** 위 숫자는 전부 코드·문서에서 직접 센 '규모' 값(자산 개수 등)이며, "얼마나 빨라졌다" 같은 추정치는 넣지 않았습니다.
- "3번 → 1번"처럼 효과를 보여 준 사례는 **대표 사례 1건**이며, 통계로 일반화한 값이 아닙니다.

> **수치 기준** — 자산 개수는 파일 수를 직접 센 값(2026-06-06)이고, '108건'은 사례 기록(반복 실수 75건 + 컨벤션 불일치 33건)의 누적입니다. 효과(시간·비용 절감 등)는 측정 체계가 없어 수치로 넣지 않았습니다.

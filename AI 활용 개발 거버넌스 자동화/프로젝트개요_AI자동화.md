# AI 기반 개발·변경 검증 자동화 — 프로젝트 개요

> 이 문서는 하네스 프로젝트 **전체**에 대한 개요입니다 (개별 문제 해결은 `문제해결_8선_AI자동화.md` 참고).
> 수치는 별도 표기가 없으면 코드와 카탈로그에서 **직접 센 실측값**이며, 측정 체계가 없는 ROI(리뷰 시간·비용·토큰 절감 등)는 추정 수치를 넣지 않았습니다.

---

## 추진 배경

자사의 클라우드 제품은 OpenStack, VMware, Kubernetes, Ansible 등을 통합 관리하는 프라이빗 클라우드 플랫폼입니다. 이 제품은 고객사별로 완전히 별도 코드베이스를 운영하는 구조가 아니라, **하나의 공통 코드베이스**를 여러 고객사가 함께 쓰고, 고객사별 차이는 plugin 구조와 설정·프로파일·외부 시스템 연동 방식으로 분리하는 구조였습니다.

고객사가 늘어남에 따라 요구사항 또한 늘어만 갔지만, 개발 인력은 같은 속도로 늘릴 수 없었습니다. 그래서 Claude Code 같은 AI 개발 도구를 도입해 한 사람이 처리할 수 있는 작업량을 늘리고자 했습니다.

> *(그림: 요구사항 증가에 따른 대응)*

문제는 **AI가 코드를 잘 만들어도, 운영 맥락을 모르면 위험하다**는 점이었습니다. 예를 들어 외부 시스템 연동에서 실제로 사용되는 값(enum), 고객사별로 분리되어야 하는 로직, 배포 파이프라인의 보호 경로, 운영 장애 경험을 통해 정착된 컨벤션은 대부분 데이터로 명확히 남아 있지 않았고, 사람이 알고 판단하던 지식이었습니다.

이런 환경에서 AI가 만든 변경을 무작정 반영하면 곧바로 문제가 발생합니다.

1. 한 고객사용 분기가 공통 영역에 섞이면 다른 고객사들 배포에 영향이 갑니다.
2. 외부 시스템(Jenkins, VMware, GitLab 등) 실제 계약과 어긋난 코드가 들어가면 운영 사고로 노출됩니다.
3. 회사 코드 컨벤션을 따르지 않으면 다른 사람이 유지보수하기 어려워집니다.
4. 이전에 발견된 오류도 사람 기억에만 의존하면 다시 반복됩니다.

그래서 이 프로젝트의 목적은 **AI가 만든 변경이 제품에 들어오기 전에 영향 범위·고객사 경계·외부 시스템 계약·회사 컨벤션을 확인하게 만들고, 한 번 발견한 오류는 다음부터 다시 발생하지 않게 만드는 것**이었습니다.

## 프로젝트 정의

이 프로젝트에서 말하는 **하네스**는 단순한 Claude 설정 파일이 아니라, AI가 코드를 변경할 때 **어디까지 허용하고 어디서 멈춰야 하는지를 정한 개발 거버넌스**였습니다.

이 프로젝트는 세 가지 설계 원칙을 기준으로 만들었습니다.

1. 사람 머릿속에 있던 운영 지식을 **규칙과 정책 파일로 꺼내는 것**
2. 자동으로 잡을 수 있는 문제는 **git hook에서 빠르게 차단하는 것**
3. 코드만으로 판단할 수 없는 외부 시스템 계약은 AI가 추측하지 못하게 **사람 확인 게이트로 돌리는 것**

> *(그림: 하네스 전체 구조)*

**첫째**, AI가 만든 코드도 사람 코드와 동일한 **3단계 검증**을 거치게 했습니다. 작성 전 영향 범위를 확인하고, 커밋 시 위험한 변경을 차단(비밀값·보호 경로는 강제 차단, 고객사 경계는 경고)했으며, 머지 후에는 안전장치와 정책 준수 여부를 다시 점검했습니다.

**둘째**, 사람과 AI가 같은 기준으로 개발하고 검토할 수 있도록 규칙·정책·검사 코드·사례를 **하나의 검증 체계로 통합**했습니다.

**셋째**, 발견된 오류는 정책과 검사 코드에 반영해 **같은 실수가 반복되지 않도록** 했습니다.

## 프로젝트 결과

이 프로젝트로 사람 머릿속에 흩어져 있던 운영 지식을 검증 가능한 형태로 꺼내고, AI가 만든 변경을 사람 코드와 같은 기준으로 검사하는 체계를 운영하게 됐습니다.

### 정량 결과

운영 중 발견된 오류·불일치 **108건**을 1회성 수정이 아니라 재사용 가능한 검증 자산 — 정책 파일 **12종**, 규칙 문서 **34종**, 자동 검사 코드 **29개** — 으로 전환했습니다. 여기에 AI 협업 자산(작업 절차서 **59개**, 역할별 작업자 **50개**)을 더해 사람과 AI가 같은 기준으로 일하게 했습니다.

- 의심 패턴 11종 중 자동 검출이 안전한 **4종(36%)** 만 커밋 직전 검사로 두고, 나머지 **7종(64%)** 은 사람 판단으로 분리
- 머지 직후 점검은 8종 중 **7종이 활성**으로 동작
- "현재 상태"를 매번 전수 측정하지 않도록, 측정 대상 17종 중 **12종(71%)** 에 유효 기간·갱신 조건을 선언
- 비밀값은 **9가지 패턴**으로 탐지하고, 로그 노출 시 **6가지 규칙**으로 가림
- 자동화 영역(`.claude/`, `scripts/ai/`) 커밋의 **82.7%** 를 단독으로 작성·운영

### 정성 결과

- AI가 영향 범위를 모른 채 "이 정도면 안전하다"고 임의 판단하고 코드 작성에 들어가던 패턴을 **차단**했습니다.
- 한 고객사용 코드가 공통 영역에 잘못 섞이는 위반을 리뷰 후반이 아니라 **커밋·CI 시점에 조기 감지**하게 됐습니다.
- 외부 시스템 계약처럼 코드만으로 알 수 없는 영역은 AI 추측 대신 사람 확인으로 돌려, 대표 사례에서 원인 파악이 **3번의 시행착오에서 1번으로(약 67%) 단축**됐습니다.
- 한 번 겪은 오류가 정책·검사로 환원되어, 다음 세션이나 다음 개발자가 **같은 실수를 반복하지 않는 구조**를 구축했습니다.
- 검증 체계 자체를 보호하는 장치(권한·보호 경로·경계 정책)를 사람 직접 승인으로만 바꿀 수 있게 해, **AI가 자기 권한을 스스로 완화하는 경로를 차단**했습니다.

### 한계 (정직하게)

리뷰 시간 절감·비용 같은 ROI 수치는 별도 측정 체계가 없어 정량화하지 못했습니다. 위 수치는 모두 코드와 카탈로그에서 직접 센 실측값이며, 측정할 수 없는 효과는 수치로 단정하지 않았습니다.

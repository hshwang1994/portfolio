# AI 기반 개발·변경 검증 자동화 — 문제 해결 8선

> 여러 고객사가 **하나의 코드베이스**를 함께 쓰는 환경에서 AI(Claude Code)로 개발 생산성을 올리되, AI가 만든 변경이 사고로 이어지지 않도록 직접 만든 검증 체계입니다. 아래는 그 과정에서 마주한 **8가지 문제**와, 각각을 **어떻게 해결했고 결과는 무엇이었는지**입니다.
>
> 각 문제의 **결과 — 무엇이 더 좋아졌나** 항목은 해결 과정을 읽지 않아도 그것만 따로 읽을 수 있게 "**그 전엔 → 이제는**" 구조로 정리했습니다. 면접에서 결과만 빠르게 보실 수 있습니다.
>
> 용어는 처음 들어도 이해되도록 풀어 썼습니다. 결과의 수치는 별도 표기가 없으면 2026-05-29 작업 브랜치에서 **코드를 직접 세어 검증한 실측값**이며, 측정 체계가 없는 ROI(리뷰 시간·비용 절감 등)는 추정 수치를 넣지 않았습니다.

---

## 문제 해결 1. 발견된 오류를 재발 방지 자산으로 만들었습니다.

**문제**
운영 중 발견한 오류를 그 자리에서만 고치면, 다음 AI 세션이나 다음 개발자가 같은 실수를 다시 할 수 있었습니다. 특히 AI는 이전 운영 사고를 기억하지 못합니다. 저장소에 보이는 코드 패턴만 보고 판단하기 때문에, 문서에 남지 않은 컨벤션이나 장애 당시의 맥락은 놓칠 수 있었습니다.

그래서 발견한 문제는 수정으로 끝내지 않고, 그 사고를 다시 발생시키지 않을 형태 — 정책 파일, 규칙 문서, 검사 코드 — 로 되돌렸습니다. 한 번 겪은 문제가 다음부터는 사람의 기억이 아니라 시스템에 의해 자동으로 막히게 만들기 위해서였습니다.

**구체적으로 어떻게 해결했는가**
발견한 사례를 먼저 "실패 사례집"에 그대로 기록했습니다. 이 기록은 추가만 되고 수정·삭제는 막아, 시간이 지나도 과거 근거가 사라지지 않게 했습니다. 그다음 사례를 성격에 따라 네 갈래로 분류했습니다 — (1) 코드 패턴으로 자동 검출 가능, (2) 변경은 감지되지만 정확한 값은 사람만 앎, (3) 코드만으론 판단 불가, (4) 잘못 잡은 것. 이 분류에 따라 데이터로 표현되는 것은 정책 파일(YAML)에, 사람 판단 근거는 규칙 문서에, 자동 검출되는 패턴은 검사 코드에 옮겼습니다. 마지막으로 옮긴 자산을 과거 사례에 다시 통과시켜 실제로 걸리는지 확인했고, 일부 검사는 과거 위반·정상 사례를 내장한 자가 테스트를 갖춰 정책을 바꿀 때 회귀를 즉시 확인하게 했습니다. 이렇게 옮긴 결과 현재 정책 12종·규칙 34종·검사 코드 29개가 운영되고 있습니다.

**결과 — 무엇이 더 좋아졌나**
**그 전엔** 발견한 오류를 그 자리에서만 고쳐, 다음 AI 세션이나 다음 개발자가 같은 실수를 반복할 수 있었습니다(AI는 과거 운영 사고를 기억하지 못합니다). **이제는** 한 번 겪은 오류가 사람의 기억이 아니라 시스템으로 막힙니다. 재발 방지 자산으로 전환한 사례가 누적 **108건**이고, 이를 정책 **12종**·규칙 **34종**·검사 코드 **29개**로 바꿔 두어 같은 사례가 다시 들어오면 자동으로 검출됩니다. 대표 예로, 발견 당시(2026-04-17) 예약 실행 코드 42개 중 **11개(26%)에만** 안전장치가 있던 상태를 자동 검사로 바꿔, 이후 같은 누락은 사람이 기억하지 않아도 잡힙니다.
(리뷰 시간 같은 정량 ROI는 측정 체계가 없어 단정하지 않습니다 — 핵심 성과는 "같은 실수를 두 번 반복하지 않는 구조"입니다.)

## 문제 해결 2. 토큰 비용과 오래된 정보 문제를 함께 줄였습니다.

**문제**
AI가 안전하게 코드를 만들려면 현재 프로젝트 상태를 알아야 했습니다. DB 구조, API 목록, 다국어 메시지 키, 화면 공통 컴포넌트, 예약 실행(스케줄러) 목록, 모듈 의존성 같은 정보가 여기에 들어갔습니다. 문제는 이 정보를 매번 처음부터 전부 읽게 하면 AI에 들어가는 정보량이 커지고 비용(토큰)이 늘어난다는 점이었습니다. 반대로 한 번 측정한 값을 계속 쓰면 오래된 정보를 기준으로 잘못된 코드를 만들 수 있었습니다.

그래서 매번 전수 측정하지도, 한 번 잰 값을 무한히 재사용하지도 않는 중간 방식을 택했습니다. 정보마다 "얼마나 자주 바뀌는가"를 기준으로 갱신 정책을 달리한 것입니다.

**구체적으로 어떻게 해결했는가**
측정 대상마다 "유효 기간"과 "다시 확인해야 하는 조건"을 미리 정해 한 곳에 선언해 뒀습니다. 예를 들어 DB 구조는 변경 스크립트가 일정 수 이상 쌓이면, API 목록은 관련 코드(컨트롤러)가 바뀌면, 화면 공통 컴포넌트는 그 컴포넌트가 수정되면 다시 확인하도록 했습니다. AI는 저장해 둔 값이 아직 유효하면 그대로 쓰고, 조건을 넘겼을 때만 그 항목 하나만 다시 측정합니다. 또 세션을 시작할 때와 다른 코드를 합친 직후에는 영향받는 측정값을 자동으로 다시 재는 장치를 둬, 사람이 신경 쓰지 않아도 최신 상태가 유지되게 했습니다.

**결과 — 무엇이 더 좋아졌나**
**그 전엔** 매번 전부 다시 읽으면 AI에 들어가는 정보량(비용)이 커지고, 한 번 잰 값을 계속 쓰면 오래된 정보로 잘못된 코드를 만들 수 있어 둘 중 하나를 포기해야 했습니다. **이제는** 보통 한쪽을 포기하는 그 둘을 함께 잡았습니다. 측정 대상 17종 중 **12종(71%)** 에 유효 기간·갱신 조건을 선언해 "변경된 항목만 다시 확인"으로 바꿔, AI에 들어가는 정보량을 줄이면서 동시에 오래된 값으로 잘못된 코드를 만드는 일도 막았습니다.
(토큰 절감률 자체는 측정 체계가 없어 수치로 단정하지 않습니다.)

## 문제 해결 3. 커밋 전에 위험 변경을 막았습니다.

**문제**
비밀값(키·토큰·비밀번호)이 코드 변경 이력에 한 번 남으면 단순 삭제로 끝나지 않았습니다. 키 교체와 이력 정리가 필요했습니다. 보호 경로(배포·CI 설정처럼 함부로 바꾸면 안 되는 파일)가 잘못 바뀌면 배포 파이프라인이나 로컬 개발 환경이 깨질 수 있었고, 한 고객사 전용 로직이 공통 영역에 섞이면 다른 고객사 배포에서 의도하지 않은 동작이 나올 수 있었습니다.

그래서 위험 변경은 코드 리뷰까지 기다리지 않고, 코드를 저장(커밋)하기 직전에 자동으로 검사했습니다. 아직 코드 이력에 들어가기 전이라 고치는 비용이 가장 낮은 시점이기 때문입니다.

**구체적으로 어떻게 해결했는가**
저장(커밋) 직전에 도는 자동 검사를 두고, 전체가 아니라 방금 바뀐 파일만 보게 해서 빠르게 동작하도록 했습니다. 다만 모든 위반을 똑같이 다루지 않고 **강제력을 두 단계로 나눴습니다.** 비밀값 노출과 보호 경로 변경처럼 돌이키기 어려운 위험은 **저장 자체를 강제로 막고**, 고객사 경계 침범처럼 코드만으로 단정하기 애매한 것은 막는 대신 **경고로 알려** 사람이 판단하게 했습니다. 잘못 막으면 개발 흐름이 끊기기 때문에, "확실한 것만 차단, 애매한 것은 경고"로 분리한 것입니다. 비밀값은 9가지 패턴(API 키·토큰·DB 접속 문자열 등)으로 탐지하고, 검출 내용이 로그에 그대로 노출되지 않도록 가렸습니다.

**결과 — 무엇이 더 좋아졌나**
**그 전엔** 비밀값 노출이나 보호 경로 훼손 같은 위험이 PR 후반이나 배포 후에야 드러나, 키 교체·이력 정리 같은 비싼 사후 복구가 필요했습니다. **이제는** 위험을 발견하는 시점이 **커밋 직전**으로 앞당겨졌습니다. 돌이키기 어려운 비밀값·보호 경로 변경은 코드 이력에 들어가기 전에 **강제로 차단**되어 사후 복구가 필요 없어졌고, 코드만으로 단정하기 애매한 고객사 경계는 차단 대신 **경고**로 남겨 과도한 차단으로 개발 흐름이 끊기는 부작용은 피했습니다. 비밀값은 9가지 패턴으로 탐지하고, 검출 내용이 로그에 노출되지 않도록 6가지 규칙으로 가립니다.

## 문제 해결 4. 머지 후 들어온 위험을 자동 보고했습니다.

**문제**
실제 개발에서는 내가 만든 코드만 저장소에 들어오지 않았습니다. 다른 개발자의 변경이 합치기(merge·pull)로 들어왔고, 그 안에 누락된 안전장치, 중복된 DB 변경 번호, 다국어 누락, 데이터 객체(DTO) 설계 오류가 함께 들어올 수 있었습니다. 내 커밋만 검사해서는 저장소 전체 상태를 지키기 어려웠습니다.

그래서 다른 코드를 합친 직후에 도는 자동 점검을 만들었습니다. 이미 합쳐진 변경을 그 자리에서 차단하기보다, 다음 작업을 시작하기 전에 위험 항목을 보고서로 드러내게 했습니다.

**구체적으로 어떻게 해결했는가**
코드를 합친 직후 자동으로 8가지 위험 패턴을 점검해 보고서를 만들도록 했습니다 — 예약 실행 코드의 예외 처리 누락, 중복된 DB 변경 번호, 다국어 키 누락·중복, 데이터 객체 오설계 등입니다. 이미 일어난 합치기는 되돌릴 수 없으므로 차단은 하지 않고, 위험을 정리할 "후속 작업 목록"으로 남겼으며, 위험도가 높은 항목은 별도 작업 티켓으로 연결했습니다. 한편 코드만으로 옳고 그름을 단정할 수 없는 항목(외부 시스템 연동 값 등)은 무리해서 자동 판정하지 않고 사람 확인 단계로 분리했습니다 — 자동 검사가 잘못 단정하면 오히려 더 큰 사고가 되기 때문입니다. 점검 항목은 8가지이고, 그중 1가지는 폐기된 규칙이라 현재 7가지가 활성입니다.

**결과 — 무엇이 더 좋아졌나**
**그 전엔** 내 커밋만 검사해서는, 다른 개발자의 변경이 합쳐져 들어올 때 묻혀 오는 위험(안전장치 누락·중복 DB 번호·다국어 누락 등)을 사람이 직접 찾아야 했습니다. **이제는** 코드를 합친 직후 7가지 위험을 자동 점검해, 합쳐 들어온 변경의 위험까지 사람 검토 전에 보고서로 드러납니다. 실제로 **129개 커밋**이 한 번에 합쳐진 대규모 머지에서도 그 안에 묻혀 있던 위반 후보를 사람이 일일이 찾기 전에 자동 점검이 먼저 잡아 후속 정리 목록으로 연결했습니다.
(구체적 검출 건수는 머지 규모와 집계 기준에 따라 달라 단일 수치로 단정하지 않습니다.)

## 문제 해결 5. 외부 시스템 계약은 AI가 추측하지 못하게 했습니다.

**문제**
외부 시스템 연동에서 가장 위험한 부분은 정답이 코드 안에만 있지 않다는 점이었습니다. 예를 들어 Jenkins(빌드·배포 도구)가 실제로 어떤 값을 받는지, VMware나 OpenStack 같은 외부 시스템을 어떤 형태로 호출하는지는 코드만 읽어서 확정할 수 없었습니다. 그런데 AI는 코드만 보고 그럴듯한 가설을 세워 진행하는 경향이 있어, 잘못된 가설을 키우며 여러 차례 시행착오를 반복했습니다.

그래서 코드 안에서 확정할 수 없는 값은 AI가 바로 구현하지 못하게 막고, 운영 환경을 아는 사람에게 먼저 묻는 절차를 강제했습니다.

**구체적으로 어떻게 해결했는가**
외부 시스템 연동 값이나 호출 방식처럼 운영 환경에서 확인해야 하는 항목은, AI가 추측으로 코드를 만드는 대신 먼저 "외부 시스템 확인 필요"로 표시하고 담당자에게 질의하게 했습니다. 담당자에게 답을 받은 뒤에만 그 값으로 구현을 진행하고, 답을 받지 못하면 결론을 "추정" 상태로 명확히 표시해 그 추정을 다른 작업의 근거(사실)로 격상해 쓰지 않도록 했습니다. 사실 이 검증은 처음에는 자동 검사로 넣었다가, "코드만 봐도 이해되는 게 좋은 코드"라는 개발자 피드백을 받고 자동 검사를 거두고 사람 질의 방식으로 바꾼 것입니다 — 자동화가 항상 정답은 아님을 인정한 결정이었습니다.

**결과 — 무엇이 더 좋아졌나**
**그 전엔** AI가 코드만 보고 그럴듯한 가설을 세워 진행하다, 외부 시스템의 실제 계약과 어긋난 잘못된 가설을 키우며 여러 차례 시행착오를 반복했습니다. **이제는** 코드로 확정할 수 없는 값을 사람에게 먼저 묻게 해, 가설을 키우며 시간을 소모하는 패턴이 사라졌습니다. 대표 사례에서 같은 유형의 원인 파악이 **3번의 시행착오에서 1번으로(약 67%) 줄었습니다**(일화적 대표 사례 기준). 코드로 알 수 없는 영역은 사람에게 맡기는 것이 더 빠르고 정확하다는 것을 보여 준 사례입니다.

## 문제 해결 6. 검사가 무한히 늘어나는 문제를 통제했습니다.

**문제**
처음에는 발견된 문제를 검사로 만들면 좋아 보였습니다. 하지만 문제마다 검사를 계속 추가하면 커밋할 때마다 대기 시간이 길어졌습니다. 또 실제 오류보다 경고가 많아지면 개발자는 피곤함을 느껴 경고를 확인하지 않고 그냥 넘기게 됩니다. 검사를 늘리는 것 자체가 오히려 검사를 무력화할 수 있었습니다.

그래서 검사를 추가하는 기준과 없애는 기준을 같이 만들어, 검사 묶음이 한 방향으로만 비대해지지 않게 했습니다.

**구체적으로 어떻게 해결했는가**
새 검사를 추가하기 전에 세 가지를 먼저 확인했습니다 — (1) 실제로 반복된 사례인가(한 번은 우연, 두 번부터 패턴), (2) 코드 패턴으로 자동 판단할 수 있는가, (3) 잘못 잡는 경우(오탐)가 개발을 방해하지 않는가. 이 기준을 통과한 것만 검사로 만들었습니다. 반대로 운영하면서 더 이상 아무것도 잡지 못하거나 경고만 쌓는 검사는 정리(보관) 대상으로 뒀습니다. 자동 판단이 어려운 항목은 무리해서 검사로 만들지 않고 사람이 직접 확인하는 단계로 남겨, 자동화와 사람 판단의 경계를 분명히 했습니다.

**결과 — 무엇이 더 좋아졌나**
**그 전엔** 문제마다 검사를 계속 추가하면 커밋 대기 시간이 길어지고, 경고가 실제 오류보다 많아지면 개발자가 결국 경고를 무시하게 되어, 검사를 늘리는 것 자체가 검사를 무력화할 수 있었습니다. **이제는** 의심 패턴 11종 중 자동 검출이 신뢰할 수 있는 **4종(약 36%)** 만 검사로 두고, 나머지 **7종(약 64%)** 은 사람이 직접 확인하도록 분리했습니다. "많이 잡는 것"이 아니라 "필요한 것만, 신뢰할 수 있게 잡는 것"을 기준으로 유지해, 검사가 무분별하게 늘어 커밋이 느려지거나 경고 과다로 개발자가 결국 무시하게 되는 상황을 막았습니다.

## 문제 해결 7. 검증 시스템 자체가 잘못 바뀌는 것을 막았습니다.

**문제**
제품 코드를 보호하는 장치가 있어도, 그 장치 자체가 잘못 바뀌면 보호 기능이 통째로 사라질 수 있었습니다. 예를 들어 보호 경로 목록에서 보호 대상을 지우거나, AI 권한 설정을 느슨하게 만들거나, 고객사 경계 정책을 약하게 바꾸면 검증 기준 전체가 약해질 수 있었습니다. 특히 AI가 자기 동작 규칙을 스스로 고칠 수 있는 구조라, 자기 권한을 스스로 풀어버리는 위험이 있었습니다.

그래서 "검증 기준 자체를 정하는 파일"은 일반 코드와 다르게 다루고, 가장 높은 위험 등급으로 분리했습니다.

**구체적으로 어떻게 해결했는가**
보호 경로 목록, AI 권한 설정, 고객사 경계 정책처럼 검증 기준을 정하는 파일을 따로 분류하고, 이들의 보호를 풀거나 권한을 완화하는 변경은 자동으로 통과시키지 않게 했습니다. 변경 위험도를 세 등급(자동 허용 / 사람 승인 필요 / 절대 자동 금지)으로 나눠, 이 보호 영역은 가장 높은 등급에 두어 **사람이 직접 사유를 남기고 승인해야만** 반영되게 했습니다. 또 검증 체계 자체의 변경은 관찰 → 설계 → 검토 → 승인 → 반영 → 재확인의 단계를 거치고, "만든 사람과 검토하는 사람을 분리"해 스스로 자기 변경을 승인하지 못하게 했습니다.

**결과 — 무엇이 더 좋아졌나**
**그 전엔** 제품 코드를 지키는 검증 장치라도 그 장치 자체(보호 경로 목록·AI 권한 설정·고객사 경계 정책)가 잘못 바뀌면 보호 기능이 통째로 사라질 수 있었고, AI가 자기 규칙을 스스로 고쳐 자기 권한을 풀어버릴 위험이 있었습니다. **이제는** 검증 기준을 정하는 이 파일들을 위험 3등급 중 최고 등급(절대 자동 금지)으로 분리해, 보호를 풀거나 권한을 완화하는 변경은 **사람이 직접 사유를 남기고 승인해야만** 반영됩니다. 설계상 이 영역은 자동으로 완화될 수 없어, AI가 자기 권한을 스스로 푸는 경로 자체가 차단됐습니다 — "검증하는 쪽"과 "검증받는 쪽"의 경계를 분명히 유지한 것입니다.

## 문제 해결 8. 멀티에이전트 작업의 환각을 줄였습니다.

**문제**
큰 작업은 하나의 AI 세션이 전부 처리하기보다 역할별 AI 작업자에게 나눠 맡기는 편이 빨랐습니다. 하지만 하위 작업자가 "완료했다"고 보고해도, 실제로는 파일이 바뀌지 않았거나, 테스트를 실행하지 않았거나, 변경 범위를 잘못 이해한 경우가 생길 수 있었습니다. 완료 보고만 믿고 다음 단계로 넘어가면 문제가 뒤에서 더 커질 수 있었습니다.

그래서 멀티에이전트 작업에서는 "완료했습니다"라는 보고만으로는 다음 단계로 넘어가지 않게 하고, 결과를 항상 실제 증거로 다시 확인하게 했습니다.

**구체적으로 어떻게 해결했는가**
작업자가 완료를 보고하면, 메인 세션이 그 말을 믿는 대신 실제 결과를 직접 확인하게 했습니다 — 실제로 어떤 파일이 바뀌었는지(코드 차이), 그 변경이 지시한 범위 안에 있는지, 테스트를 돌렸고 통과했는지, 남은 오류가 없는지를 다시 봤습니다. 또 작업자가 "아마", "추정" 같은 불확실한 표현으로 보고하면 그 결론을 확정 사실로 격상하지 않게 했습니다. 한 작업자에게 한 번에 너무 많이 맡기면 응답이 잘려 실패하므로, 한 작업자당 2~3개 파일 규모로 잘게 나눴습니다(예: 21개 파일이면 7명에게 3개씩).

**결과 — 무엇이 더 좋아졌나**
**그 전엔** 하위 AI 작업자가 "완료했다"고 보고해도 실제로는 파일이 바뀌지 않았거나 테스트를 돌리지 않은 경우가 있어, 완료 보고만 믿고 다음 단계로 넘어가면 문제가 뒤에서 더 커졌습니다. **이제는** 완료를 "말"이 아니라 실제 코드 차이·변경 범위·테스트 통과 여부로 다시 확인해, 파일을 만들지 않고 "완료"라고 한 환각 보고를 걸러냅니다. 작업은 한 작업자당 2~3개 파일 규모로 잘게 나눠(예: 21개 파일이면 7명에게 3개씩) 병렬로 빠르게 처리하면서도, 잘못된 완료 보고 위에 다음 작업을 쌓아 사고가 커지는 것을 막았습니다.

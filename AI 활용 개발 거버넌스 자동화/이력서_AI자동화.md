# AI 기반 개발 및 변경 검증 자동화

| 항목 | 내용 |
|------|------|
| 기간 | 2026.04 ~ 현재 |
| 참여인원 | 자동 검증 시스템은 본인 단독 설계·구현 (제품 저장소 전체는 팀 협업) |
| 역할 | 자동 검증 시스템 구조 설계, 검증 규칙 작성, 자동 실행 스크립트 구현, 개발자 피드백 반영한 정책·규칙 지속 개선 |
| 기술스택 | Python(표준 라이브러리만 사용 — 외부 의존성 최소화), Git Hook, YAML, Bitbucket Pipeline |

## 추진 배경

여러 고객사(고객사 A~F)의 요구사항이 계속 늘어났습니다. 별도로 인력을 충원하는 대신 AI 도구(Claude Code 등)를 활용해 개발 생산성을 높이는 방향을 골랐습니다.

다만 이 제품은 한 코드베이스를 여러 고객사가 함께 쓰고, 고객사마다 자원 관리 방식이 다릅니다. AI가 만든 결과를 그대로 반영하면 한 고객사용 코드가 다른 고객사 배포에 섞여 들어가거나, 외부 시스템(Jenkins, VMware, GitLab 등) 연동 기준과 어긋난 채 운영 사고로 이어질 수 있습니다. 코드가 회사 컨벤션을 따르지 않으면 다른 사람이 유지보수하기 어려워집니다.

그래서 AI를 활용해 개발하는 사람이 안전하게 쓸 수 있도록, 개발 단계부터 커밋, 머지까지 단계별로 자동 검사하는 안전장치를 직접 만들었습니다. AI가 코드 작성에 들어가기 전에 영향 범위와 컨벤션 준수를 확인하고, 코드 저장(커밋)과 머지 직후에도 자동으로 검사합니다.

## 해결 과정

- 고객사 코드 경계 자동 감지 — 한 고객사용 분기가 공통 영역에 섞이면 다른 고객사들 배포까지 영향을 줍니다. 여러 고객사가 각각 어떤 파일을 쓰는지 매핑을 정책 파일(YAML)에 정리하고, 코드를 저장(커밋)하거나 CI가 돌 때 경계 위반을 자동으로 검사해 경고합니다. (시크릿이나 보호 경로 침범은 저장 자체를 막고, 고객사 경계 위반은 경고로 알려 사람이 판단하도록 했습니다.)

- 개발 단계 영향 검토와 컨벤션 검토 게이트 — AI가 영향 범위를 모른 채 "이 정도면 안전하다"고 임의 판단하면 나중에 재작업이 발생합니다. AI가 회사 코드 컨벤션을 모른 채 코드를 만들면 사람이 다시 다듬어야 합니다. 코드 변경 요청("추가해줘", "리팩토링" 등)이 감지되면 영향 받는 모듈, 고객사, 리스크 등 5개 항목과 회사 코드 컨벤션(빈 다국어 메시지 키(i18n), 문자열 검증 함수 오용(hasLength/hasText), 미완료 표시 주석(TODO) 같은 의심 패턴)을 먼저 확인하도록 정책 레벨로 안내합니다. AI는 이 단계를 스스로 건너뛸 수 없고, 사용자가 명시적으로 'skip'을 지시할 때만 생략됩니다.

- 다른 개발자 변경 자동 점검 보고서 — 다른 개발자나 AI가 만든 코드가 합쳐질 때(머지) 검토 없이 묻히면 운영 사고로 이어집니다. 합쳐진 직후 스케줄러 예외 처리 누락, DB 마이그레이션 번호 중복 같은 위험 패턴을 자동으로 점검해 보고서로 정리합니다. 외부 시스템 연동값 불일치처럼 코드만으로는 옳고 그름을 판단할 수 없는 영역은, 무리해서 자동 점검하지 않고 담당자에게 먼저 확인하는 절차로 분리했습니다.

- Closed-loop(자기개선) 구조 — 운영 중 발견된 정책 위반과 컨벤션 어긋남을 사례 카탈로그와 감사 로그(머지 점검 보고서, 자산 변동 스냅샷)에 기록하고, 그 사례를 다시 검증 규칙과 자동 점검 스크립트로 옮겨 같은 문제가 다시 들어오면 자동으로 잡히게 했습니다. 구조나 정책을 바꾼 판단 근거는 ADR(의사결정 기록)로 따로 남겨, 무엇을 왜 검출하도록 정했는지 추적할 수 있습니다. 이렇게 **발견 → 기록 → 규칙화 → 자동 검출**로 순환시켜, 같은 문제를 두 번 반복하지 않는 자기개선(Closed-loop) 구조를 만들었습니다.

## 결과

- **제품 개발 재작업 3건 → 1건 (66.7% 감축)** — 외부 연동 오류를 코드 추측 대신 사람 선확인으로 전환한 대표 사례 기준
- **AI 활용 개발 오류 108건을 정책화** → 같은 오류가 다시 들어오면 자동으로 검출되는 재발 방지 체계 구축

운영 중 발견된 오류와 고객사 코드 경계 위반 사례를 누적 **108건**까지 모았습니다. 이를 분석해 재사용 가능한 검출 자산 — 정책 파일(고객사·경계 데이터) **12종**, 검증 규칙 문서(검출 의도·근거) **34종**, 자동 실행 스크립트(자동 검출 로직) **29개** — 으로 정리했습니다(여러 사례가 하나의 규칙·검사로 합쳐지므로 사례 수와 자산 수는 1:1이 아닙니다). 그 결과 같은 유형의 문제가 다시 들어오면 자동으로 검출됩니다.

도입 후 측정 가능한 효과:

- 개발 단계에서 AI 임의 판단 차단 — 예전에는 AI가 영향 범위를 모른 채 "이 정도면 안전하다"고 자체 판단해서 코드 작성에 들어갔습니다. 이제는 영향 범위와 회사 코드 컨벤션을 먼저 확인해야만 작성을 시작할 수 있어, AI가 만든 코드도 사람이 직접 만든 코드와 같은 기준으로 검토됩니다.

- 경계 위반 발견이 빨라짐 — 한 고객사용 코드가 공통(shared) 영역에 잘못 들어가는 위반은 예전에는 코드 리뷰 후반에 다른 고객사 담당자가 직접 찾아냈습니다. 이제는 AI와 함께 만든 코드를 저장하거나 CI가 도는 시점에 자동으로 감지돼, 리뷰 후반까지 가기 전에 드러납니다.

- 외부 시스템 연동 오류 원인 파악이 빨라짐 — 예를 들어 코드에 정의된 Jenkins 작업 종류 목록이 실제 Jenkins 지원값과 어긋난 경우입니다. 대표 사례(2026-04-23 Jenkins 작업 종류 불일치)에서 AI가 코드만 보고 추측하느라 **3회 시행착오**를 거쳤습니다. 외부 시스템 담당자에게 먼저 확인하는 절차로 바꾸고 나서는 같은 사례를 **1회**에 파악합니다.

- 대량 머지에서도 빠짐없이 점검 — 한 번에 129건의 작업 결과가 합쳐진 적이 있었습니다. 그 안에 들어 있던 잠재 위반 **11건**을 사람 검토 전에 자동 보고서가 잡아냈습니다.

- 새 고객사 추가 비용 감소 — 예전에는 새 고객사가 들어올 때마다 검증 코드를 직접 수정해야 했습니다. 이제는 정책 파일에 한 블록만 추가하면 즉시 자동 검증 대상에 포함됩니다.

# SK하이닉스 인프라 자동화 플랫폼 개발

| 항목 | 내용 |
|------|------|
| 기간 | 2026 ~ 현재 |
| 참여인원 | 플랫폼 아키텍처 설계와 핵심 구현을 본인이 단독 주도 |
| 역할 | 폐쇄망 인프라 자동화 플랫폼 아키텍처 설계, 포털 자동화 구동 표준 정의, 멀티벤더 서버 정보 개더링 파이프라인 작성 |
| 기술스택 | CI/CD는 Jenkins, 자동화는 Ansible, 저장소는 GitLab과 Nexus, 점검 프로토콜은 Redfish/vSphere/SSH/WinRM, fact 캐싱은 Redis, 언어는 Python을 썼습니다 |

## 추진 배경

신규 반도체 클러스터가 들어오면서 관리할 인프라가 빠르게 커졌습니다. 장비를 준비하는 시간, 운영 인력 비용, 반복 수작업에서 나오는 휴먼 에러를 줄이려고 인프라를 코드로 다루는 자동화 플랫폼을 만들었습니다. 환경은 까다로웠습니다. 인터넷이 없는 폐쇄망이고, GitLab과 Jenkins는 자체 고가용성이 없는 무료 오픈소스였으며, 개발망과 운영망이 갈리고 데이터센터도 지역 A, B, C로 떨어져 있었습니다.

## 해결 과정

- **플랫폼 기반을 폐쇄망에 세움** — Jenkins, Ansible, Nexus 오픈소스 기반 인프라 자동화 아키텍처를 설계해, SK하이닉스가 인프라를 코드로 직접 관리하도록 사내 클라우드 포털에 통합했습니다. 무료 오픈소스라 자체 HA가 없어, 이중화를 인프라 레벨로 풀었습니다. GitLab은 VMware FT, Jenkins 마스터는 /data1 공유 스토리지 기반 active-passive입니다.

- **포털에서 자동화를 구동** — 포털에서 선택한 자산을 자동화 실행 대상으로 바로 넘기려고, Jenkins로 전달된 호스트 JSON을 동적 인벤토리로 구성해 Ansible 실행 대상에 반영했습니다.

- **IP 하나로 정보 개더링** — IP만 입력하면 서버 유형을 자동 판별해 하드웨어(Redfish), OS, 가상화(ESXi) 정보를 수집하고, 자산으로 등록하도록 표준 JSON으로 통일하는 Ansible 파이프라인을 만들었습니다.

- **리전과 러너 확장을 스크립트화** — 신규 리전과 Jenkins 러너를 빠르게 늘리도록 Ansible 실행 환경 설치 과정을 스크립트로 만들었습니다.

- **구형 Linux 대응** — Python 기반 Ansible 모듈이 실패하는 구형 Linux는 raw 명령 기반 수집 경로를 따로 구성해 같은 표준 형식으로 수집했습니다.

- **반복 점검 단축** — 동일 호스트를 반복 점검할 때 OS 정보 재수집을 Redis 캐시로 생략해 수집 시간을 줄였습니다.

## 결과

- 포털 클릭 한 번으로 멀티벤더 서버를 점검하고 운영하는 자동화 플랫폼을 세웠습니다. 이 플랫폼으로 2026년 7월 체결을 목표로 14.9억 규모 본계약을 추진 중입니다(체결 전 추진 단계).
- 폐쇄망 플랫폼 기반은 Round 8.1까지 검증했고, 구축과 운영 자동화에 shell 스크립트 47개, Jenkins 파이프라인 4개, Ansible playbook 7개를 작성했습니다.
- 포털 자동화 구동은 동적 인벤토리로 호스트 수와 무관하게 같은 경로로 처리하고, 그 영역 커밋 58개 중 57개(98.3%)를 본인이 작성했습니다.
- 정보 개더링은 IP 하나로 3채널을 점검해 13필드 표준 JSON으로 돌려주고, 공개 GitHub 커밋 335개 중 327개(97.6%)를 본인이 작성했습니다. Redfish 수집 엔진은 표준 라이브러리만으로 3,812줄입니다.
